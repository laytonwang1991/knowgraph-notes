# 推理服务架构

## 一句话定义

推理服务架构是支撑大语言模型（LLM）高效推理的完整系统设计方案，通过优化计算调度、内存管理和推理引擎，实现高吞吐、低延迟的模型部署。

## 详细说明

### 1. LLM推理服务系统设计

LLM推理服务系统需要解决三个核心问题：

- **延迟（Latency）**：首Token生成时间决定用户体验
- **吞吐（Throughput）**：系统每秒能处理的请求数
- **成本（Cost）**：GPU资源利用效率

传统推理架构采用静态批处理，但无法充分利用GPU。现代架构引入连续批处理（Continuous Batching）和分页注意力（PagedAttention）实现资源高效利用。

### 2. vLLM推理引擎

vLLM是由伯克利大学开发的开源推理引擎，核心创新是PagedAttention技术。

#### 核心特性

- **PagedAttention**：借鉴操作系统虚拟内存管理思想，将KV Cache分页存储
- **连续批处理**：动态将新请求加入正在执行的批处理
- **异步执行**：请求入队和推理执行异步进行
- **FlashAttention集成**：高效注意力计算

#### 架构设计

```

```

### 3. TensorRT-LLM

TensorRT-LLM是NVIDIA开发的推理优化引擎，针对NVIDIA GPU深度优化。

#### 优化技术

- **算子融合（Kernel Fusion）**：减少内存访问
- **量化支持（INT8/FP8）**：减少计算量和内存占用
- **投机解码（Speculative Decoding）**：加速自回归生成
- **张量并行（Tensor Parallelism）**：多卡并行推理

#### 性能对比

| 引擎 | Throughput (tokens/s/GPU) | 内存效率 |
|------|---------------------------|----------|
| naive HF | 50 | 低 |
| vLLM | 200+ | 高 |
| TensorRT-LLM | 300+ | 最高 |

### 4. 连续批处理（Continuous Batching）

连续批处理解决了静态批处理的两个问题：
1. 新请求必须等待当前批次完成
2. 不同序列长度导致GPU闲置

#### 实现原理

```python
class ContinuousBatchingScheduler:
    def __init__(self, max_batch_size=32):
        self.running_batches = []
        self.pending_requests = deque()
        self.max_batch_size = max_batch_size

    def step(self):
        # 1. 回收已完成的序列
        self._collect_finished_sequences()

        # 2. 尝试加入新请求
        while (len(self.running_batches) < self.max_batch_size
               and self.pending_requests):
            req = self.pending_requests.popleft()
            self._add_to_batch(req)

        # 3. 执行一步推理
        self._execute_batch()
```

### 5. 分页注意力（PagedAttention）

PagedAttention将KV Cache存储在非连续内存块中，类似于操作系统的分页管理。

#### 核心思想

- 将KV Cache分成固定大小的块（如64个token一块）
- 每个序列的KV Cache由多个块组成，通过Block Table映射
- 消除预留最大长度内存的浪费

#### 内存效率提升

传统方式：为每个请求预留`max_length * hidden_size`内存
PagedAttention方式：按实际使用的token分配内存

内存节省：**可达60-80%**

## 代码示例

### vLLM服务部署

```python
from vllm import LLM, SamplingParam

# 初始化推理引擎
llm = LLM(
    model="meta-llama/Llama-2-70b-hf",
    tensor_parallel_size=4,  # 使用4张GPU
    gpu_memory_utilization=0.9,
    max_num_batched_tokens=8192,
    max_num_seqs=256,
)

# 推理调用
sampling_params = SamplingParams(
    temperature=0.8,
    top_p=0.95,
    max_tokens=512,
)

outputs = llm.generate(["Hello, world!", "Write a poem."], sampling_params)

for output in outputs:
    print(f"Generated: {output.outputs[0].text}")
```

### 自定义调度器

```python
class PriorityScheduler:
    def __init__(self, llm_engine):
        self.engine = llm_engine
        self.queue = PriorityQueue()

    def add_request(self, prompt, priority=1):
        """priority: 1-10, 越高越优先"""
        request_id = self.engine.add_request(prompt)
        self.queue.put((priority, request_id))

    def step(self):
        if not self.queue.empty():
            priority, req_id = self.queue.get()
            self.engine.step(request_id=req_id)
```

## 应用场景

### 1. 在线推理服务

- **ChatGPT类应用**：需要低延迟的首Token响应
- **实时翻译**：P95延迟<500ms要求
- **代码补全**：如GitHub Copilot，需要<100ms延迟

### 2. 离线批处理

- **文档处理**：大量PDF/文章的批量总结
- **数据标注**：大规模AI辅助标注
- **内容生成**：批量文章、营销文案生成

### 3. 多模态推理

- **视觉问答**：图像+文本的联合推理
- **视频理解**：长视频的逐帧分析
- **语音合成**：流式输出降低感知延迟

### 4. 企业级部署

- **私有化部署**：数据安全要求高的场景
- **多租户系统**：不同客户的模型隔离
- **混合精度推理**：平衡精度和性能

## 相关概念

| 概念 | 说明 |
|------|------|
| Prefill阶段 | 首次计算，将prompt编码并生成首个token |
| Decode阶段 | 自回归生成后续token |
| KV Cache | 存储注意力机制的Key-Value矩阵 |
| Beam Search | 束搜索，保留多个候选序列 |
| Speculative Decoding | 投机解码，用小模型预测大模型输出 |
